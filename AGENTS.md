# AGENTS.md

## ⚠️ 强制性指令

**收到任何任务后，第一步必须读取本文件并执行任务分类。**

---

## 项目说明

本项目采用 Harness Engineering。所有任务必须严格遵循 `.harness` 目录中的工作流程。

---

## 核心原则

### 强制性流程入口

任何任务开始前，必须执行以下步骤：

1. **任务分类**：分析用户意图，判断任务类型
2. **流程选择**：读取对应 workflow 文档
3. **检查清单**：读取对应 checklist 文件
4. **步骤执行**：按照 workflow 步骤顺序执行，不得跳过
5. **阶段输出**：每个阶段完成后输出明确的进展

---

## 任务分类系统

### 判断方法

任务分类基于**意图分析**，而非仅关键词匹配：

1. **精确匹配**：用户明确说明任务类型（如"帮我修复一个bug"）
2. **意图推断**：分析用户的核心目标和问题类型
3. **默认流程**：无法判断时，使用 feature workflow

### 任务类型对照表

| 任务类型 | 核心意图 | 触发场景 | 对应 Workflow |
|---------|---------|---------|---------------|
| **新功能** | 添加新能力 | 创建、实现、新增 feature | `.harness/workflows/feature.md` |
| **Bug 修复** | 解决已有问题 | 错误、异常、崩溃、不工作 | `.harness/workflows/bugfix.md` |
| **重构** | 改善代码结构 | 优化、改进、整理、重写 | `.harness/workflows/reactor（重构）.md` |
| **代码审查** | 检查代码质量 | review、审查、检查、评估 | `.harness/workflows/review.md` |
| **架构设计** | 设计系统方案 | 架构、设计、方案、系统 | `.harness/workflows/architecture（架构）.md` |
| **技术调研** | 探索技术方案 | 调研、研究、探索、评估 | `.harness/workflows/research.md` |

### 混合任务处理

当任务涉及多种类型时：

1. **确定主类型**：以用户的核心目标为基准选择主 workflow
2. **标注次类型**：在流程中标注涉及的次要类型
3. **交叉读取**：读取主 workflow 后，还需读取次类型对应的 rules/checklist
4. **阶段适配**：在合适阶段引入次要类型的检查项

## 流程入口规范

### 第一步：任务分类输出

每次收到用户请求后，**必须**输出：

```
【任务分类】
- 识别结果：[任务类型]
- 判断依据：[用户意图分析]
- 对应流程：[workflow 文件路径]
- 对应检查：[checklist 文件路径]

是否需要调整任务类型？
```

### 第二步：读取 Workflow

确认任务类型后，**必须**读取对应的 workflow 文档，理解并输出流程步骤。

### 第三步：读取 Checklist

读取对应 checklist 文件，确保每个步骤都有检查点：

| 任务类型 | Checklist 文件 |
|---------|---------------|
| 新功能 | `.harness/checklist/feature.md` |
| Bug 修复 | `.harness/checklist/bugfix.md` |
| 重构 | `.harness/checklist/refactor.md` |
| 代码审查 | `.harness/checklist/review.md` |
| 前端任务 | `.harness/checklist/frontend.md` |

### 第四步：流程自检

每个阶段开始前，**必须**输出：

```
【流程自检】
- 当前阶段：[阶段名称]
- 上一阶段：[已完成/未完成]
- 检查项：[当前阶段 checklist]
```

### 第五步：执行流程

按照 workflow 文档中的步骤顺序执行。

---

## 强制性输出要求

### 流程启动时

输出：
- 任务类型判断结果
- 将要执行的流程名称和文件路径
- 流程步骤概览

### 每个阶段完成后

输出：
- 当前阶段名称
- 完成的工作内容
- 下一阶段计划

### 任务完成时

输出：
- 修改的文件列表
- 验证结果
- 潜在问题和注意事项

## 上下文控制原则

### 最小必要上下文

阅读文档之前，请先判断：

1. 我是否已经知道答案？
2. 当前代码是否已经足够？
3. 是否真的需要额外文档？

只有答案为"需要"时，再读取对应文档。

### 文档读取策略

#### workflow

```
.harness/workflows/
```

- 新功能 → feature.md
- 修复 Bug → bugfix.md
- Code Review → review.md
- 重构 → reactor（重构）.md
- 架构设计 → architecture（架构）.md
- 技术调研 → research.md

#### rules

```
.harness/rules/
```

只有涉及对应技术栈时才读取：
- React
- Next.js
- TypeScript
- Git
- 前端性能
- 代码质量

#### skills

```
.harness/skills/
```

只有需要使用某项能力时才读取：
- Three.js
- Prisma
- Supabase

#### knowledge

```
.harness/knowledge/
```

只有出现以下情况时才读取：
- 不清楚项目架构
- 不确定技术选型
- 不理解历史设计决策

## 优先级

解决问题时，遵循以下优先级：

1. 当前用户需求
2. 当前代码
3. Harness 文档
4. 自身经验

## 最终目标

始终保持：
- 最少读取
- 最少修改
- 最少上下文
- 最大正确性
- 严格流程
