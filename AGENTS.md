# AGENTS.md

## 项目说明

本项目采用 Harness Engineering。

`.harness` 目录不是需要全部读取的知识库，而是按需检索的工程规范。

你的目标不是阅读尽可能多的文档，而是在保证正确性的前提下，以最小上下文完成任务。

---

## 工作原则

开始任何任务前，请遵循以下原则：

- 优先利用当前对话提供的信息。
- 优先阅读当前正在修改的代码。
- 不要主动遍历整个 `.harness`。
- 只有当缺少必要信息时，再读取对应 Harness 文档。
- 每次只读取当前任务真正需要的内容。

---

## Harness 结构

`.harness` 包含以下内容：

- workflows：不同类型任务的工作流程
- rules：项目开发规范
- skills：特定技术或框架的最佳实践
- knowledge：项目长期知识（架构、技术栈、设计决策）

---

## 文档读取策略

### workflow

当你不知道当前任务应该如何推进时，再阅读：

```
.harness/workflows/
```

例如：

- 新功能 → feature.md
- 修复 Bug → bugfix.md
- Code Review → review.md
- 重构 → refactor.md

---

### rules

只有当涉及对应技术栈时，再阅读：

```
.harness/rules/
```

例如：

- React
- Next.js
- TypeScript
- Git

如果当前任务与某项技术无关，则不要读取对应规则。

---

### skills

只有当需要使用某项能力时，再阅读：

例如：

- Prisma
- Supabase
- BullMQ
- TanStack Query

不要提前加载所有 Skill。

---

### memory

只有出现以下情况时，再阅读：

- 不清楚项目架构
- 不确定技术选型
- 不理解历史设计决策

否则优先从代码理解项目。

---

## 上下文控制

始终遵循：

**最小必要上下文（Minimal Context）**

不要因为存在 Harness 而全部读取。

阅读文档之前，请先判断：

1. 我是否已经知道答案？
2. 当前代码是否已经足够？
3. 是否真的需要额外文档？

只有答案为"需要"时，再读取对应文档。

---

## 优先级

解决问题时，请遵循以下优先级：

1. 当前用户需求
2. 当前代码
3. Harness 文档
4. 自身经验

不要因为 Harness 存在，而忽略已有代码实现。

---

## 最终目标

始终保持：

- 最少读取
- 最少修改
- 最少上下文
- 最大正确性
